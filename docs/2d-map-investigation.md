# WoWMap 2D 化地图调查

## 调查目标

本次调查要回答的问题不是“怎么画一张魔兽地图”，而是：

- 客户端里是否存在可直接用于 2D 导航底图的官方数据
- 哪些数据能保证地图上的距离和游戏世界尽量一致
- V1 应该先做什么层级的 2D 化，才能最快跑通导航

## 已确认的数据来源

### 1. `WorldMapArea.dbc`

这是目前最重要的 2D 化基础数据。

它已经提供了：

- 大陆级范围
- 区域级范围
- 游戏世界坐标和地图显示坐标之间的映射边界

从现有解析结果看：

- `map_id=0` 对应东部王国
- `map_id=1` 对应卡利姆多
- 大陆记录本身已经给出了 `X/Y` 范围
- 各个区域也带有自己的二维范围矩形

这意味着：

- 我们可以先建立一套可信的 `游戏坐标 -> 2D 坐标` 线性映射
- 两个大陆可以分别建立自己的 2D 平面
- 交通节点、区域中心、区域矩形都可以稳定投影到前端

### 2. `WorldMapContinent.dbc`

这个文件提供的是大陆级的偏移和缩放辅助信息。

目前看它更像 UI 世界地图的布局参数，而不是完整边界来源。

它有价值，但优先级低于 `WorldMapArea.dbc`：

- 可作为大陆级校验参考
- 可帮助理解 Blizzard UI 的显示方式
- 但单独靠它不足以完成导航底图

### 3. `WorldMapOverlay.dbc`

这个文件更像地图覆盖层和标签块信息，不是精确区域边界。

当前观察到的记录更接近：

- 某些区域名对应的覆盖块
- 覆盖块的矩形大小
- 与 area id 的关联

因此它适合做：

- 标签补充
- 区域显示辅助
- 后续的简化区域轮廓参考

但它不适合直接作为：

- 精确多边形边界
- 真正的导航轮廓数据

### 4. `Interface\\WORLDMAP\\*.blp`

客户端里存在：

- `Interface\\WORLDMAP\\Azeroth\\*.blp`
- `Interface\\WORLDMAP\\Kalimdor\\*.blp`

这些是地图贴图瓦片，说明客户端 UI 世界地图本质上是栅格图，不是现成向量图。

它们可以用于：

- 人工对照
- 调试投影是否正确
- 验证交通节点落点

但不适合作为 V1 的核心地图库，因为：

- 它们本质上还是图片
- 不满足我们“尽量轻量、尽量向量化”的目标

### 5. `Interface\\WORLDMAP\\Azeroth.zmp` / `Kalimdor.zmp`

客户端中还存在 `.zmp` 文件，但目前还不能证明它们能直接产出导航需要的几何边界。

现阶段不建议把 V1 建立在这类文件上，除非后续确认它们承载的是明确可解析的地理几何数据。

## 当前最重要的结论

客户端里能直接支持的是：

- 大陆和区域的二维范围
- 世界坐标到地图平面的投影基础

客户端里暂时不能直接保证的是：

- 精确的区域向量边界
- 可直接拿来用的世界多边形轮廓

所以 V1 的 2D 化不应该定义成“先得到完整向量地图”，而应该定义成：

先建立一个与游戏坐标一致的二维导航平面。

## 推荐的 V1 2D 化方案

### 第一层：真实比例坐标底座

先以大陆为单位建立 2D 平面：

- 东部王国一个平面
- 卡利姆多一个平面

规则：

- 前端坐标必须从原始游戏坐标线性变换得到
- 允许平移、缩放、坐标轴翻转
- 不允许为了美观做局部拉伸

这样可以保证：

- 地图上的相对距离可信
- 任意点点击可以回映到游戏世界坐标
- 路线计算和显示共用同一套空间基准

### 第二层：区域级简化表达

V1 不要求精确多边形边界。

更稳妥的做法是先用这些对象表示地图：

- 大陆边界
- 区域范围矩形
- 区域中心点
- 主城位置
- 飞行点
- 船 / 飞艇节点
- 传送节点

这已经足够支撑：

- 地图选点
- 区域识别
- 最近交通节点接入
- 路线结果展示

### 第三层：后续再补简化轮廓

如果后面需要更像地图，而不是只是导航面板，可以再逐步补：

- 简化后的大陆轮廓
- 简化后的区域轮廓

这些轮廓应当服务于可读性，不应破坏坐标比例。

## 为什么 V1 不该先追求完整向量边界

原因很简单：

- 客户端现成可取的是坐标范围，不是高质量 polygon
- 如果一开始执着于边界还原，项目会立刻回到高投入、低回报路线
- 导航真正依赖的是坐标和交通图，而不是精美区块轮廓

换句话说：

V1 的关键不是“地图像不像原版”，而是“点下去算出来的路线准不准”。

## 对 WoWMap 的设计建议

V1 应该采用“节点优先、轮廓从简”的 2D 地图方案。

具体建议：

1. 先把东部王国和卡利姆多做成两个真实比例平面
2. 先投影交通节点，而不是先画复杂边界
3. 先用区域矩形或简化区块作为点击和识别基础
4. 先让地图能选点并算路
5. 边界美化和轮廓精修放到后续阶段

## 建议的下一步

1. 把 `WorldMapArea.dbc` 解析结果导出成可复用的中间数据
2. 为东部王国和卡利姆多生成一份最小二维底图数据
3. 把已有交通节点投影到这个平面上
4. 做一个只显示节点和区域框的最小网页原型
5. 验证“地图点选 -> 最近交通节点 -> 算路结果”这条链路是否跑通

## 当前判断

2D 化地图在 V1 的正确落点不是“向量美术生产”，而是“建立一个可信的二维导航坐标系”。

只要这个坐标系成立，后面的路线规划、地图点击和结果展示才会稳。
