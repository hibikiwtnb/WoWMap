# WoWMap

WoWMap 是一个面向 `World of Warcraft 3.3.5` 的轻量导航工具。

项目目标不是复刻游戏大地图美术，而是做一个足够可靠、足够轻量、能真正帮助玩家规划路线的地图系统。它应该像一个魔兽世界版的 Google Maps：支持在地图上选点、计算跨交通方式的路线，并尽量发现玩家平时不容易想到的高效率走法。

## V1 目标

- 只支持 `WLK 3.3.5`
- 只覆盖 `东部王国 + 卡利姆多`
- 地图使用抽象、轻量的向量表达
- 地图比例尽量对齐游戏 3D 世界实际距离
- 提供一个最小可用网页
- 支持在地图上选择起点和终点
- 支持 `任意点到任意点` 和 `区域到区域` 的路线规划
- 优先实现 `时间最短` 路线

## V1 支持范围

### 支持的交通方式

- 步行
- 坐骑移动
- 飞行点
- 船
- 飞艇
- 传送门
- 职业自力移动能力

### 暂不支持

- 炉石
- 死亡跳楼 / 墓地复活捷径
- 依赖队友的移动能力
- 地形阻挡下的真实地面寻路
- 高保真地图美术

## 当前设计前提

### 产品定位

WoWMap 是导航工具，不是世界探索器。

它的第一优先级是帮助玩家节省时间，甚至找出看起来反直觉、但总耗时更低的路线。

### 起终点规则

- 起点和终点默认来自地图点击
- V1 至少支持：
  - 任意点到任意点
  - 区域到区域
- 任意点接入路由图时，先连接到同区域内最近的可用交通节点

### 区域内移动模型

- V1 允许用直线距离估算区域内步行或骑乘成本
- 不考虑山体、墙体、水域等真实绕路
- 后续可按角色是否有坐骑切换速度模型

### 职业能力

V1 只考虑角色自己可以独立完成的位移能力，优先考虑：

- 法师传送 / 传送门
- 德鲁伊月光林地传送
- 死骑自力传送能力

## 数据原则

- 当前客户端路径：`C:\Games\wlk_335`
- 优先使用官方客户端数据
- 可接受高可信服务端数据作为补充
- 可接受 WOWHead 等外部资料作为校验与补缺
- 可将原始数据解析为本项目自己的 JSON / 中间格式
- 可基于官方坐标推导时间、距离、接入成本等字段
- 不接受不可追溯、不可重建的人工拼装数据作为主数据源

建议数据可信度优先级：

1. 客户端官方数据
2. 高可信服务端数据
3. WOWHead / 社群资料

## 系统拆分

项目初步拆成四个子系统：

1. `extractors`
   - 提取客户端、DBC、服务端数据中的原始地图和交通信息
2. `normalizer`
   - 统一坐标系、区域定义、节点类型、边类型
3. `routing`
   - 构建 travel graph 并执行最短时间路线规划
4. `viewer`
   - 提供一个最小网页界面用于选点和展示结果

## V1 路由模型

一次完整规划分成三段：

1. 起点接入交通图
2. 在交通图中跨方式规划
3. 从交通图退出到终点

这意味着系统需要同时处理两类边：

- 交通边：飞行点、船、飞艇、传送门等离散交通连接
- 接入边：任意点或区域接入最近交通节点的步行 / 骑乘成本

## 数据建模原则

- 在没有真正完成客户端或服务端数据解析前，不提前固化字段名
- 尽量沿用原始数据中的命名与语义，避免建立第二套容易冲突的词汇体系
- 只有在跨来源统一时，才为中间层增加最小必要的映射字段
- 中间层的职责是规范化，不是重新发明一套完全独立的数据结构
- 设计重点先放在“实体关系”和“数据流向”，而不是字段命名

当前可以先从概念层面理解系统至少需要表达以下几类信息：

- 区域与大陆边界
- 可作为导航锚点的交通节点
- 节点之间的交通连接关系
- 任意点接入交通网络时的估算移动成本
- 路线查询输入条件
- 路线结果分段输出

## 坐标系统提醒

这条提醒非常重要：

客户端数据里的地图坐标方向，和玩家在游戏中实际看到的地图方向，不是同一套朝向。

也就是说，直接把客户端里的二维范围或节点坐标按常规方式画出来，方向会不对。

### 当前项目中的已确认现象

在 WoWMap 当前原型里，如果想让地图朝向和游戏里看到的方向一致，显示层需要做额外修正。

目前已经验证过的显示修正结果是：

1. 逆时针旋转 90 度
2. 左右镜像

做完之后，地图上的朝向才和游戏中的观察结果一致。

### 这条提醒的意义

后续开发者在处理这些数据时，需要先知道一件事：

- 客户端坐标是可信的
- 但它们的显示朝向，不等于玩家在游戏界面里看到的朝向

所以如果后面有人发现：

- 地图整体方向怪怪的
- 南北看起来对了，但左右反了
- 节点位置大致对，但视觉方向不对

这通常不是数据坏了，而是显示坐标系还没有做对齐。

### 当前相关位置

当前这个方向对齐逻辑在：

- `app.js` 的显示投影函数

后面如果重构前端或改投影方式，必须记得重新验证这件事，否则很容易再次在这里踩坑。

## 第一阶段验收标准

- 能用结构化数据表示东部王国与卡利姆多的核心交通网络
- 能从地图点击生成起点和终点
- 能输出一条按时间排序的可解释路线
- 能展示分段结果，例如步行、飞行、船、传送门的切换

## 当前结论

这个项目的成功关键，不在于先做界面，而在于先建立一条可信、可重建、可验证的数据链路。

只有当底层交通图可靠，地图界面才有意义。
